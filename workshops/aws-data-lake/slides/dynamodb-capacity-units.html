WCU, RCU, WRU, RRU — Four Units, Two Modes

Same physical work, different billing model depending on capacity mode

UnitModeWhat One Unit Buys
WCUProvisioned1 write/second, up to 1 KB (pre-provisioned, billed hourly whether used or not)
RCUProvisioned1 strongly-consistent read/second, up to 4 KB — or 2 eventually-consistent reads/sec for the same unit
WRUOn-demandOne write request, up to 1 KB — billed per request actually made
RRUOn-demandOne strongly-consistent read request, up to 4 KB
⚠️ Rounding rule: items larger than the per-unit size round up. A 2.5 KB write costs 3 WCU/WRU. A 10 KB strongly-consistent read costs 3 RCU/RRU (10 KB → rounds to 12 KB ÷ 4 KB).
Key Point Provisioned = pay for capacity you reserve, whether you use it. On-demand = pay per request, no reservation.Source: DynamoDB throughput capacity

rWCU / rWRU: The Global Tables Gotcha

AWS's own worked example — the moment a table goes global, the "origin" write is no longer free

⚠️ Common misconception: that only the "other" Region pays a replicated unit. Actually — once a table has even one replica, every write in every replica Region (including the one you wrote to) bills as rWCU/rWRU. There is no free origin write on a global table.
Day 1
Single-Region table in us-west-2. Write 100 items (1 KB each).
100 WRUsnormal, single-Region
Day 2
Add a replica in us-east-1 — table is now global. Write 150 items (1 KB each) to the us-west-2 replica.
300 rWRUs150 in EACH Region
Day 3
Add a GSI (auto-created on both replicas). Write 200 more items.
400 WRUs + 400 rWRUsGSI writes ≠ rWRU
Same price
rWCU/rWRU cost identical to WCU/WRU per unit — the extra cost is paying once per Region, not a price premium
GSI ≠ replicated
GSI updates always bill as plain WCU/WRU, in every Region that has the index
Reads stay local
RCU/RRU are only charged where the read request lands — replication is a write-side cost model only
Teaching Point Day 1's 100 writes cost 100 units total. Day 2's 150 writes cost 300 units total — the per-write cost roughly doubled the moment the table became global with 2 Regions, purely from each Region billing its own copy.Source: Understanding Amazon DynamoDB billing for global tables
