Three Data Store Categories, Three Different Jobs

Every enterprise platform ends up with all three — the question is never "which one wins," it's "which job is this"

Relational (OLTP)

Aurora / RDS

  • Structured data in tables, rows, columns
  • Supports complex queries through joins
  • Best fit for less predictable query patterns
  • Migrating/modernizing on-prem relational workloads
Non-Relational (OLTP)

DynamoDB / DocumentDB

  • Key-value, document, and other flexible models
  • Optimized for large volumes, low latency
  • Relaxes some consistency for scale + speed
  • Session stores, catalogs, high-throughput apps
Analytical (OLAP)

Redshift

  • Fully managed cloud data warehousing
  • Handles large-scale analytics workloads
  • Store and analyze massive data quickly
  • Historical, aggregate, cross-source queries
Key Distinction AWS's own database decision guide splits databases into OLTP (relational + non-relational, this slide's left two cards) and OLAP — "to store and analyze massive amounts of data quickly and efficiently... AWS offers Amazon Redshift."Source: AWS Decision Guide — Choosing an AWS database service

A Framework for the DynamoDB / DocumentDB / RDS / Redshift / Athena Decision

Six questions that narrow five services down to one

Access pattern
Known key lookups → DynamoDB. Rich document queries → DocumentDB. Ad-hoc joins → RDS/Aurora. Cross-source aggregate queries → Redshift or Athena.
Schema flexibility
Fixed, well-known schema → relational. Evolving / semi-structured attributes per item → DynamoDB or DocumentDB.
Latency requirement
Single-digit millisecond at any scale → DynamoDB. Seconds-to-minutes for large scans → Redshift/Athena are fine.
Query complexity
Multi-table joins, transactions → relational. Single-item or single-collection operations → NoSQL. Multi-way historical joins → Redshift.
Data freshness for analytics
Query data in place, infrequently, no infra to manage → Athena over S3. Frequent, high-concurrency BI → Redshift.
Operational scale
Millions of requests/sec, unpredictable traffic → DynamoDB on-demand. Steady, predictable enterprise workload → RDS/Aurora.
Honest Framing AWS does not publish one page that ranks DynamoDB vs. DocumentDB vs. RDS vs. Redshift vs. Athena side by side — this criteria list is an editorial synthesis built from each service's own documented strengths. Tell students they're synthesizing, not citing a single source.

One Real Platform, Three Databases at Once

AWS's own worked example of the framework in practice — an e-commerce platform

DocumentDB

  • Product catalogs
  • User profiles

Chosen for the flexibility of semi-structured data — catalog attributes vary by product category.

DynamoDB

  • Product catalog browsing

Chosen for low, predictable latency while users are actively browsing — the highest-traffic, most latency-sensitive path.

Aurora

  • Inventory and order processing

Chosen because this workload needs a relational data model and transaction support — orders must be atomic and consistent.

Turning every table mutation into an action, without polling anything

How AWS Documents This

  • You associate a stream's ARN with a Lambda function you write — a trigger
  • All mutation actions to the table can be captured as an item on the stream
  • The Lambda function can send a notification, initiate a workflow, or copy each stream record to persistent storage (e.g. S3) as a permanent audit trail

AWS's Own Example: GameScores → Lambda → Social Post

The exact scenario AWS uses to explain DynamoDB Streams + Lambda triggers

The Scenario, As AWS Documents It

  • A mobile gaming app writes to a GameScores table
  • Whenever the TopScore attribute of the table is updated, a corresponding stream record is written to the table's stream
  • This event triggers a Lambda function that posts a congratulatory message on a social media network
  • The function is written to ignore any stream records that are not updates to GameScores, or that do not modify TopScore

DocumentDB Change Streams: CDC for the Document World

Same idea as DynamoDB Streams, MongoDB-compatible API

What They Capture

  • A time-ordered sequence of change events within a cluster's collections
  • All change events supported by the MongoDB watch() API — insert, update, delete, replace
  • Must be explicitly enabled per collection, per database, or cluster-wide

Documented Use Cases

  • Change notification to downstream applications
  • Full-text search with Amazon OpenSearch Service
  • Analytics with Amazon Redshift

Fitting NoSQL Into the Raw → Enriched → Consolidated Pipeline

DynamoDB and DocumentDB aren't a separate silo — they're another CDC source feeding the same pipeline as Days 1-4

Raw

Landing Zone

  • DynamoDB Streams / Kinesis Data Streams for DynamoDB → S3
  • DocumentDB change streams → S3 (via a consumer app or Lambda)
  • Immutable, as-captured change events
Enriched

Transform Zone

  • Glue ETL / Lambda merges change events into curated tables
  • Schema conformance, dedup, before/after image handling
Consolidated

Serving Zone

  • Redshift for BI / historical joins
  • Athena over S3 for ad-hoc query-in-place
  • OpenSearch for full-text / operational search
Course Framing "Raw → Enriched → Consolidated" is this program's own naming for the zone pattern — the underlying AWS concept (bronze/silver/gold, or raw/curated/serving) is standard, but this exact terminology is ours.

Security Integration: Same Three Pillars as Day 3

IAM, VPC, and KMS apply to DynamoDB and DocumentDB exactly as they applied to everything else this program has covered

IAM

  • Centrally manage who can access which tables/collections
  • No usernames/passwords for DynamoDB access — IAM only
  • Fine-grained access control down to the attribute level

VPC

  • Gateway endpoints and Interface endpoints (PrivateLink) for DynamoDB
  • No public internet exposure required for in-VPC access
  • DocumentDB clusters live inside a VPC by design

KMS

  • All DynamoDB data encrypted at rest by default
  • Three key types: AWS owned, AWS managed, customer managed
  • Switch key types on an existing table without app changes
Key Point There is no separate "NoSQL security model" to learn — it's the identical IAM/VPC/KMS toolkit from Day 3, applied to two more services.
