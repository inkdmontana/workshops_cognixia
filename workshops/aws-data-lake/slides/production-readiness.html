Three-Zone S3 Data Lake — Enterprise Standard

Separation of concerns between raw, curated, and serving data — plus three enterprise additions

Zone 1 — Bronze

RAW

  • Exact copy of source data as received
  • Immutable — never overwrite, never transform in place
  • Format: CSV, JSON, binary — as-landed
  • Retention: matches source SLA (7 years regulated)
  • Partitioned by ingest date only
Access: pipeline service roles only — no analysts
Zone 2 — Silver

CURATED

  • Validated, deduplicated, schema-conformed
  • Format: Parquet, snappy-compressed, partitioned
  • Partitioned by: year / month / business key
  • Schema enforced here — no raw inconsistencies pass
  • Registered in Glue Catalog, Lake Formation governed
Access: data engineers + approved pipeline roles
Zone 3 — Gold

SERVING

  • Aggregated, business-ready, domain-specific
  • Loaded into Redshift for heavy analytics
  • Available via Athena for ad-hoc, Spectrum for joins
  • Lake Formation row/column filters applied
  • BI-tool and analyst-facing
Access: analysts, BI tools, data scientists
🚧 Quarantine BucketFailed quality gate records — visible, tracked, actionable
🧊 Archive BucketS3 Glacier — raw data beyond active retention window
📋 Audit BucketCloudTrail + Lake Formation logs — WORM-protected, no delete
Key Insight The quarantine bucket is what separates mature platforms. Bad data landing in curated silently poisons every downstream consumer. Data landing in quarantine is visible, tracked, and actionable. No quarantine bucket = no production platform.

Security Hardening — Encryption at Every Layer

Data must be encrypted at rest and in transit everywhere. No exceptions. No "it slows down Glue."

Encryption at Rest (KMS)

Amazon S3
  • SSE-S3 (free) for non-regulated data
  • SSE-KMS: CMK + every decrypt audited in CloudTrail
  • DSSE-KMS: dual-layer — required by some government/financial frameworks
  • Bucket policy: Deny s3:PutObject where encryption header not set
Amazon Redshift
  • Cluster-level KMS encryption — enable at creation
  • Cannot add retroactively — requires snapshot + restore
  • Redshift Serverless: encrypted by default
AWS Glue + CloudWatch Logs
  • Glue Catalog metadata encryption with KMS
  • Job bookmark encryption (also KMS)
  • CloudWatch log group encryption: KMS key on the group

KMS Key Management Rules

  • One CMK per classification tier — not one key for everything
  • Automatic annual rotation on all CMKs
  • Key policy: restrict kms:Decrypt to specific roles — not *
  • Cross-account: grant to specific principal, not all external

Encryption in Transit (TLS)

S3 — Force HTTPS
  • Bucket policy with Deny on aws:SecureTransport: false
  • Blocks any HTTP (unencrypted) request to the bucket
  • Apply to raw, curated, serving, quarantine, and audit buckets
Amazon Redshift
  • Parameter group: require_ssl = true
  • TLS 1.2 minimum for all client connections
  • JDBC/ODBC drivers enforce certificate validation
All AWS APIs (Athena, Glue, Step Functions)
  • HTTPS enforced by AWS on all service endpoints
  • Athena results land in S3 — inherit bucket encryption
  • Secrets Manager: all credentials stored encrypted, never in code
Instructor Note — "Encryption slows Glue"
  • SSE-S3 has near-zero performance impact
  • SSE-KMS adds latency only on the key call — which is cached
  • The performance argument for skipping encryption is not valid in 2026

Security Hardening — VPC, IAM & Service Control Policies

Network isolation · least privilege · org-level preventive controls

VPC Isolation

Glue jobs in VPC
Without VPC assignment, Glue jobs can reach the public internet. Always assign a VPC + subnet.
S3 VPC Gateway Endpoint
Glue / Lambda → S3 traffic stays on the AWS backbone. No NAT Gateway. No public path. Free.
Redshift in private subnet
No public IP. Accessed via VPC only. Security group: inbound port 5439 from BI subnet only. No 0.0.0.0/0.
Interface endpoints
Glue, Athena, STS, KMS, Secrets Manager — all via VPC endpoints. Eliminates NAT Gateway cost and public traffic.

IAM Least Privilege

Glue job role
s3:GetObject on raw prefix only + s3:PutObject on curated prefix only. Not s3:*.
Lambda quality gate role
cloudwatch:PutMetricData + s3:GetObject on specific prefix. Nothing else.
Redshift role
s3:GetObject for COPY + glue:GetTable for Spectrum. No write permissions to data.
No access keys, ever
All services use IAM roles. No long-lived access keys. No inline policies. Managed policies in version control.

Service Control Policies

Region lockdown
Deny all actions where aws:RequestedRegion not in approved list (us-west-2, us-east-1).
Encryption mandate
Deny s3:PutObject where s3:x-amz-server-side-encryption not set.
Audit trail protection
Deny cloudtrail:StopLogging and cloudtrail:DeleteTrail — no principal can silence the audit trail.
No public S3
Deny s3:PutBucketPublicAccessBlock with BlockPublicAcls=false. Public buckets cannot be created.
Production OU — Strictest SCPsEncryption mandatory · region locked · no direct console data access · no IAM admin roles without boundaries
Non-Production OU — RelaxedEngineers can experiment · no public S3 still enforced · CloudTrail still required · encryption still required

Infrastructure as Code — CloudFormation vs CDK

If it is not in code, it does not exist. A click-created platform cannot be reproduced, reviewed, or trusted.

CloudFormation

What it is
  • AWS-native IaC — declarative JSON or YAML
  • Direct 1:1 mapping to AWS resource properties
  • Supports all services, day-one for new AWS features
  • Drift detection: live state vs template → flags manual changes
  • Stack sets: deploy across multiple accounts and regions simultaneously
Best for
  • IAM roles and policies
  • S3 bucket policies and lifecycle rules
  • KMS keys and key policies
  • CloudWatch alarms, dashboards, SNS topics
  • Stable, low-change platform infrastructure

CDK (Cloud Development Kit)

What it is
  • Write infrastructure in Python, TypeScript, Java, Go
  • Compiles to CloudFormation — same deployment engine underneath
  • L2 constructs: new s3.Bucket(this, 'Raw', { encryption: BucketEncryption.KMS })
  • L3 constructs (patterns): entire pipeline as one reusable construct
  • Better for teams already writing Python/TypeScript
CDK vs CloudFormation — the truth
  • CDK does NOT replace CloudFormation — it synthesizes CF templates
  • All drift detection, rollback, stack management = CloudFormation
  • CDK and CloudFormation are complementary, not competing
  • Pick by team language preference, not by feature capability
Acid Test: Delete your entire AWS environment. Re-create it from code in under 2 hours. If you cannot — your IaC is incomplete. This is not hypothetical: it is your disaster recovery test.

What Goes in IaC for a Data Platform

S3 buckets + policies + lifecycle
All IAM roles + managed policies
Glue databases + crawlers + job defs
KMS keys + key policies
CloudWatch alarms + dashboards
SNS topics + subscriptions
Step Functions state machines
Lake Formation permissions

Change Management — Deploying Data Services Safely

Data services are harder to deploy than web services — schema changes break consumers, and you cannot roll back written data

CI/CD Pipeline — The Five Stages

1
Lint + Unit Test
Lambda/Glue script passes syntax checks and unit tests against mock data
2
Deploy to Staging
CloudFormation/CDK deploys to staging environment. Same SCPs, same VPC config, no real PII
3
Integration Test
Invoke pipeline end-to-end. Validate output row count and schema match expected
4
Manual Approval Gate
Required for regulated industries. Automated prod deploys without human sign-off fail SOC2 CC8.1
5
Deploy + Monitor
Deploy to production. Watch CloudWatch dashboard 15 minutes. Auto-rollback if error alarm fires within 30 min

Schema Evolution Safety Rules

SAFEAdd a nullable column — existing queries unaffected
SAFEAdd a new partition key going forward — old partitions still readable
UNSAFERename a column — breaks all existing queries immediately
UNSAFEChange a column type — silent breakage (widening) or loud (narrowing)
UNSAFERemove a column — immediate breakage for all consumers

Blue/Green for Glue Jobs

  • Blue = current production job version
  • Green = new version deployed alongside blue
  • Run green against staging dataset, validate output
  • Switch Glue trigger to green after validation
  • Keep blue running for 24 hours as rollback
  • Breaking schema change? New table version (v2) + migration window

Data Quality — The Four Gate Types

Quality is enforced before data moves downstream — not discovered after analysts complain

1Schema Validation
  • Do all expected columns exist with correct names?
  • Are column types correct? (string where date expected)
  • Are required columns present at all?
  • Implemented: at Lambda ingestion entry point
On fail: route to quarantine, emit SchemaMismatch=1, alarm fires
2Null & Type Checks
  • Required fields: customer_id, trade_date, price — must not be null
  • Type check: numeric fields contain no "N/A" strings
  • Range check: price > 0 AND < 1,000,000
  • Referential: region in [NA, EMEA, APAC, LATAM]
On fail: quarantine + NullViolationCount + RangeViolationCount metrics
3Row Count Reconciliation
  • Source row count (from object metadata tag) vs loaded rows
  • Delta threshold: abs(source − loaded) / source > 0.1%
  • Pipeline ran, SUCCEEDED, wrote 0 rows — this gate catches it
  • Lab 3: RowCountDelta metric → CloudWatch alarm
On fail: quarantine + RowCountDelta metric → ALARM (missing=breaching)
4Pipeline Heartbeat
  • Lambda Invocations < 1 for 2 expected run intervals → ALARM
  • Lambda Duration > SLA threshold → ALARM
  • S3 object not created in expected prefix within 5 min → ALARM
  • Catches: pipeline did not run (not just wrong output)
On fail: alert immediately — no data = worse than wrong data
Quarantine Pattern Raw → Quality Gate Lambda → PASS → Curated (clean)    → FAIL → Quarantine (tagged with failure reason)

Lambda Quality Gate — Three Checks & Custom Metrics

Architecture: S3 PutObject event → Lambda → CloudWatch custom metric → Alarm → SNS

1
Schema Check
  • Read Parquet header, extract column names and types
  • Compare against schema in SSM Parameter Store
  • On mismatch: tag object ingest-status=quarantined, move to quarantine prefix, publish SchemaMismatch=1, return
2
Null & Range Check
  • Load first 1,000 rows — not entire file for large datasets
  • For each required column: assert null count = 0
  • For each numeric column: assert min > lower AND max < upper
  • Publish NullViolationCount and RangeViolationCount
3
Row Count Reconciliation
  • Read expected count from S3 object metadata tag (set by producer at write time)
  • Compare against actual rows in file
  • Compute delta as a percentage
  • If delta > threshold: quarantine + alarm
  • Publish RowCountDelta metric

CloudWatch Custom Metrics

Namespace: DataPlatform/QualityGate
SchemaMismatchCount (0=pass, 1=fail)
NullViolationCountCount per file
RangeViolationCountCount per file
RowCountDeltaPercent
QualityGatePassRatePercent (5-min window)

Alarms

SchemaMismatch > 0 → ALARM immediately (1/1 period)
RowCountDelta > 0.1% → ALARM, missing=breaching
QualityGatePassRate < 99% → ALARM if sustained 2 periods

Key Pattern

  • Producer writes expected row count as S3 object tag at write time
  • Gate reads tag at validation time — decouples from re-querying source
  • No source system connection required in the quality Lambda

Performance Benchmarking — Athena vs Spectrum vs Redshift Direct

For a given workload: which engine gives the right performance at the right cost? Run the math — don't guess.

Amazon Athena

$5 / TB scanned
  • Serverless SQL on S3 — no cluster, no provisioning
  • Parquet + partitions = less scan = lower cost
  • 20 concurrent DML queries per account (soft limit)
  • Results delivered to S3 — fetch after completion
Best: ad-hoc, infrequent large scans, unpredictable workloads
Worst: high-concurrency dashboards, sub-second SLAs

Redshift Spectrum

$5/TB S3 scan + node hours
  • Redshift SQL engine reads S3 via external tables
  • JOIN S3 data with Redshift internal tables in one query
  • No separate ETL required for cross-source joins
  • Requires an active Redshift cluster to use
Best: S3 data + Redshift dimension table JOINs
Worst: teams without Redshift (paying nodes + scan)

Redshift Direct

Node hours only — no scan charge
  • Data physically loaded via COPY into columnar storage
  • Pre-sorted, compressed, distributed — fastest for repeated queries
  • High-concurrency with WLM queue management
  • Concurrency Scaling: auto spin-up on demand
Best: dashboards, sub-second SLAs, repeated aggregations
Worst: cold historical data queried quarterly
Infrequent queriesOnce or rarely → Athena
Daily / hourlyRepeated patterns → Redshift Direct
S3 + Redshift JOINCross-source in one SQL → Spectrum
Sub-2s SLAHigh concurrency → Redshift Direct
Pay per queryUnpredictable volume → Athena
Pay for capacityPredictable, high-freq → Redshift
Hot + cold data90-day in RS, historical S3 → Hybrid Lakehouse
Join needed + S3Avoid extra ETL → Spectrum

Concurrency Testing & Cost-Per-Query Crossover

The engine that works for 1 user may fail for 50 — always benchmark concurrency before committing to an architecture

Concurrency Testing Protocol

Athena Concurrency Test

  • Submit 20 queries simultaneously via boto3
  • Measure: StartQueryExecution → SUCCEEDED per query
  • Watch for: queries queuing at the 20-query account limit
  • Mitigation: request limit increase, or shard across workgroups
  • Cost signal: Athena Workgroup per-query scan limit prevents runaway scans

Redshift Concurrency Test

  • Submit from 50 simultaneous sessions (pgbench or JMeter)
  • CloudWatch: WLMQueueLength > 0 during test = saturation
  • Watch: WLMQueueWaitTime — time waiting, not executing
  • Tune: increase WLM concurrency slot count for analyst queue
  • Nuclear option: Concurrency Scaling — spins additional clusters

Cost-Per-Query Example — Work This Out Live

Scenario: Daily sales aggregation · 500 GB scan · 8 runs/day
Athena cost per run 500 GB × $5/TB = $2.50
Athena cost per day $2.50 × 8 = $20
Athena cost per month $20 × 30 = $600
Redshift RA3.xlplus 2-node cluster $0.482 × 2 × 730 = $704/mo
Redshift cost per run (this query only) $704 ÷ 240 runs = $2.93
If 20 queries share cluster $704 ÷ 4,800 = $0.15/run
Crossover logic: This query alone → Athena cheaper ($600 vs $704). Twenty queries share the cluster → Redshift wins by 16×. The crossover point: when total monthly Athena spend > Redshift cluster cost, load into Redshift.

Common wrong answer: "put everything in Redshift." 7 years of historical data queried 4×/year → Athena at $20/year vs Redshift storage at $2,760/year. Run the math out loud every time.

Operational Runbooks — Structure That Works at 3 AM

Written for the engineer who just got paged and has never touched this pipeline — not for the engineer who built it

Section 1

Alarm & Trigger

Alarm name, metric, threshold, and evaluation period. No interpretation required.

quicklabs-RowCountDelta-too-high
RowCountDelta > 0.1% · 1/1 period · 5 min
Section 2

Impact Statement

What breaks. Who is affected. SLA implications.

Curated sales data is stale.
Reporting dashboards show yesterday's figures.
End-of-day reporting pipeline at risk.
Section 3

Immediate Triage (<5 min)

  • CloudWatch → Alarm History → when did it first fire?
  • S3 → quarantine bucket → files today?
  • Lambda → Monitor → error count last hour?

Assess before acting. Do NOT fix anything yet.

Section 4

Diagnosis (5–15 min)

  • Exact SQL or CLI command — no improvisation
  • Exact CW log group + filter pattern
  • Exact console navigation path (no "go look in the console")
Section 5

Remediation Steps

  • Numbered — no ambiguity about order
  • Exact action at each step
  • Fallback if Step N fails
  • Escalation: page [name] if not resolved in 30 min
Sections 6–7

Verify + Post-Incident

  • Specific check proving fix worked
  • Metric returned to OK (don't silence — verify)
  • Jira ticket: timeline, root cause, action items
  • Update runbook if steps were wrong
Runbook Quality Test Have a junior engineer execute the runbook against a staged failure — without help. If they cannot complete it in the target time, the runbook is incomplete. Runbooks rot — version-control them alongside the IaC they describe.

Incident Response — Five-Phase Pattern

The most common failure: jumping to Phase 4 (fix) without Phase 2 (triage). Fixing the wrong thing costs 30+ minutes.

Phase 1
Detect
0–2 min
  • CloudWatch Alarm → SNS → PagerDuty / email → on-call acknowledges
  • Open war room: Slack channel + Zoom. Declare severity: P1 (down), P2 (degraded), P3 (data quality)
Phase 2
Triage
2–10 min
  • Infrastructure or data? — Lambda not running / Glue failed / Redshift unhealthy vs output wrong or late
  • Isolated or systemic? — one file = schema issue; all files = IAM permission or upstream source change
  • Check Slack for analyst complaints before assuming you found it first
Phase 3
Mitigate
10–30 min
  • Goal: stop the bleeding, NOT fix the root cause
  • Glue infinite retry → disable trigger, let job exhaust. Bad data reached curated → revoke Lake Formation access.
  • Redshift overloaded → kill runaway query (pg_cancel_backend), raise WLM concurrency
  • Communicate status every 10 minutes — even with no new information
Phase 4
Resolve
30 min–4 hrs
  • Fix root cause. Re-run failed pipeline with corrected code or corrected input.
  • Verify with row count reconciliation. Restore Lake Formation access. Mark alarm as OK (verify, don't silence).
Phase 5
Review
Within 24 hrs
  • Blameless post-mortem: timeline, root cause, contributing factors
  • Five Whys until you hit a systemic gap — not "be more careful"
  • Action items with owners and due dates. Update runbook, alarm, test suite.

Production Readiness Checklist

Rate each item: DONE · IN PROGRESS · NOT STARTED · N/A — "Done" means linked evidence exists (IaC file, test result, runbook doc)

Infrastructure

  • All resources in CloudFormation or CDK
  • Multi-region CloudTrail + log integrity validation
  • S3: versioning on, MFA delete, public access blocked
  • All sensitive buckets encrypted SSE-KMS
  • KMS: rotation on, least-privilege policy, one key per tier
  • SCPs: region lock, encryption mandate, no CloudTrail disable
  • Glue in VPC with S3 endpoint. No long-lived access keys.

Data Layer

  • Raw / Curated / Serving zone separation
  • Quarantine bucket configured
  • Schema validation gate on ingestion
  • Null / range / type checks on required fields
  • Row count reconciliation for all batch loads
  • Lake Formation column security on PII columns
  • S3 lifecycle: Glacier transition + expiry on all buckets

Pipeline

  • All pipelines in Step Functions (or orchestrator)
  • Retry: exponential backoff, max 3 retries
  • DLQ on all SQS queues used by Lambda
  • Heartbeat alarm: fires if pipeline misses run window
  • CI/CD: test → staging → manual approval → prod
  • Blue/green tested for at least one critical Glue job

Observability

  • Dashboard: CPU, disk, WLM, Lambda errors, Athena bytes
  • Alarm: Redshift disk > 85%
  • Alarm: Lambda error rate > 5% (not raw count)
  • Alarm: WLMQueueLength > 0 sustained 2 periods
  • Alarm: SchemaMismatch, RowCountDelta
  • Log retention set on all log groups (no never-expire)
  • GuardDuty on in all regions, S3 Protection enabled

Cost

  • Cost allocation tags activated in Billing
  • Budget alert at 80% and 100% of monthly target
  • Cost Anomaly Detection for Lambda + Redshift
  • Athena workgroup scan limit set
  • Redshift usage limit: Spectrum + Concurrency Scaling
  • Compute Optimizer enabled + recommendations reviewed

Runbooks & Recovery

  • Runbook for every P1 and P2 alarm
  • Runbooks tested by junior engineer in last 90 days
  • Redshift snapshot restore tested successfully
  • S3 versioning: accidental delete recovery tested
  • RTO defined and tested (Recovery Time Objective)
  • RPO defined (max acceptable data loss)

Knowledge Check — KC-15

Self-grade · 10 minutes · Mix of recall and judgment

Q1 · Recall
A Glue job writes 0 rows to the curated bucket but returns status SUCCEEDED. Which quality gate would have caught this?
  • A) Schema validation
  • B) Null check on required fields
  • C) Row count reconciliation
  • D) Pipeline heartbeat alarm
Answer: C — Row count: 0 loaded vs 50,000 expected = gate fires. Schema and null checks cannot catch empty output. Heartbeat catches "did not run," not "ran and wrote nothing."
Q2 · Judgment
200 analyst queries/day, 500 GB each, 3-second SLA. Athena costs $3,200/month. 2-node Redshift RA3 = $1,400/month. Should you migrate?
  • A) No — Athena is serverless and simpler
  • B) Yes — Redshift will be cheaper and faster
  • C) Benchmark 200 concurrent queries at peak load first
  • D) Use Redshift Spectrum instead
Answer: C — Math suggests Redshift wins ($1,400 vs $3,200), but you must benchmark 200 concurrent queries to verify the 3-second SLA holds before committing.
Q3 · Recall
Which SCP condition prevents S3 objects from being written without encryption?
  • A) Deny s3:GetObject where encryption header is missing
  • B) Deny s3:PutObject where s3:x-amz-server-side-encryption not set
  • C) Deny s3:CreateBucket where encryption not enabled
  • D) Deny s3:PutBucketPolicy where policy allows public access
Answer: B — Must target the write action (PutObject) and check the encryption request condition. GetObject is a read — encryption is irrelevant at read time.
Q4 · Judgment
On-call gets paged for a RowCountDelta alarm at 2 AM. Checks S3 quarantine — no files there. What should they do next?
  • A) Assume transient glitch and go back to sleep
  • B) Check if the Lambda quality gate itself failed to run
  • C) Re-run the pipeline immediately
  • D) Escalate to senior engineer without further investigation
Answer: B — Empty quarantine + alarm = the gate may have crashed before writing anywhere. Check Lambda error logs. An unhandled exception writes to neither curated nor quarantine.
Q5 · Recall
Which statement about CloudFormation and CDK is correct?
  • A) CDK replaces CloudFormation — you do not need CloudFormation if you use CDK
  • B) CDK compiles down to CloudFormation templates and uses the same deployment engine
  • C) CloudFormation supports Python; CDK supports only JSON and YAML templates
  • D) CDK is only suitable for serverless applications, not data platforms
Answer: B — CDK synthesizes CloudFormation templates. Deployment, rollback, drift detection, and stack management all happen via CloudFormation under the hood. CDK and CloudFormation are complementary — not competing.
